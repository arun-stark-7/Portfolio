Authentication
PayFlow uses API keys to authenticate requests. Include your key in every request using the Authorization header.

Request header

Authorization: Bearer YOUR_API_KEY
Content-Type: application/json
Never expose your API key in client-side code or public repositories.

API key types

Key Type Prefix Environment Description
Live Key pf_live_ Production Real transactions
Test Key pf_test_ Sandbox No real charges
Restricted Key pf_rstr_ Both Limited permissions

Example cURL request

curl -X GET https://api.payflow.io/v2/payments \
  -H "Authorization: Bearer pf_live_xK9mN2qR7vB4" \
  -H "Content-Type: application/json"

Create a Payment
POST /payments
Creates a new payment transaction. Amounts are in the smallest currency unit (cents for USD).

Request parameters

ParameterTypeRequiredDescription
amount integer Required Amount in smallest unit (e.g. 1000 = $10.00)
currency string Required ISO 4217 currency code (USD, EUR, GBP)
payment_method string Required Token from PayFlow.js tokenization
customer_id string Optional Associate with an existing customer
description string Optional Arbitrary description (max 500 chars)
metadata object Optional Key-value pairs for additional data
capture boolean Optional If false, only authorizes (default: true)

Request body example

{
  "amount": 4999,
  "currency": "USD",
  "payment_method": "pm_xK9mN2qR7vB4wL8j",
  "customer_id": "cus_P3tY3cZ6nM1kQ9wR",
  "description": "Pro Plan - Monthly Subscription",
  "metadata": { "order_id": "ORD-20250601-4821" }
}

Response example — 200 OK

{
  "id": "pay_7vB4wL8jxK9mN2qR",
  "object": "payment",
  "amount": 4999,
  "currency": "USD",
  "status": "succeeded",
  "customer_id": "cus_P3tY3cZ6nM1kQ9wR",
  "created_at": "2025-06-01T10:23:45Z",
  "description": "Pro Plan - Monthly Subscription"
}

Status codes

200 Payment created and captured successfully
202 Authorized but not captured (when capture: false)
400 Bad request — invalid or missing parameters
401 Unauthorized — invalid or missing API key
402 Payment required — card declined or insufficient funds
422 Unprocessable — payment method invalid or expired
429 Too many requests — rate limit exceeded
500 Internal server error — contact support
Retrieve a Payment
GET /payments/{id}
Retrieves the details of an existing payment using its unique identifier.

Path parameters

ParameterTypeRequiredDescription
id string Required Unique payment ID (e.g. pay_7vB4wL8jxK9mN2qR)

Example request

curl https://api.payflow.io/v2/payments/pay_7vB4wL8jxK9mN2qR \
  -H "Authorization: Bearer pf_live_xK9mN2qR7vB4"
Refund a Payment
POST /payments/{id}/refund
Issues a full or partial refund for an existing payment. Refunds appear on your customer's statement within 5–10 business days.

Request parameters

ParameterTypeRequiredDescription
amount integer Optional Amount to refund in cents. Defaults to full amount.
reason string Optional duplicate, fraudulent, or customer_request
Refunds can only be issued within 90 days of the original payment.
Create a Customer
POST /customers
Creates a new customer object. Use customers to track payments, subscriptions, and saved payment methods.

Request parameters

ParameterTypeRequiredDescription
email string Required Customer's email address
name string Optional Full name of the customer
phone string Optional Customer's phone number
metadata object Optional Key-value pairs for additional data

Example request

{
  "email": "priya@example.com",
  "name": "Priya Sharma",
  "phone": "+91-98765-43210",
  "metadata": { "plan": "enterprise" }
}
Update a Subscription
PATCH /subscriptions/{id}
Partially updates a subscription. Only fields included in the request body will be changed.

Request parameters

ParameterTypeRequiredDescription
plan_id string Optional ID of the new plan to switch to
quantity integer Optional Number of seats or units
cancel_at_period_end boolean Optional If true, cancels at end of billing cycle
metadata object Optional Key-value pairs for additional data
Error Handling
PayFlow returns standard error objects. Always check the HTTP status code and parse the error body to handle failures gracefully.

Error object structure

FieldTypeRequiredDescription
error.code string Required Machine-readable code (e.g. card_declined)
error.message string Required Human-readable error description
error.param string Optional Parameter that caused the error
error.doc_url string Optional Link to error-specific documentation

Error response example — 402

{
  "error": {
    "code": "card_declined",
    "message": "Your card has insufficient funds.",
    "param": "payment_method",
    "doc_url": "https://docs.payflow.io/errors/card_declined"
  }
}

Common error codes

CodeHTTPDescription
card_declined 402 Card was declined by the issuer
insufficient_funds 402 Card has insufficient funds
expired_card 422 Payment method has expired
invalid_api_key 401 API key is missing or invalid
rate_limit_exceeded 429 Too many requests in the current window
invalid_currency 400 Currency code is not supported
Rate Limiting
PayFlow enforces rate limits per API key. Limit information is returned in every response as HTTP headers.

Rate limit headers

HeaderTypeDescription
X-RateLimit-Limit integer Max requests allowed per minute
X-RateLimit-Remaining integer Requests left in current window
X-RateLimit-Reset integer Unix timestamp when window resets
Retry-After integer Seconds to wait (only on 429 responses)

Limits by plan

PlanRequests / minuteRequests / day
Starter10010,000
Growth500100,000
Enterprise2,000Unlimited
If you receive a 429 response, wait for the number of seconds in the Retry-After header before retrying.
